# Release v0.1.5 - Infisical Kubernetes Auth (Option C)

**Release Date**: 2026-03-02  
**Branch**: `v0.1.5`  
**Status**: In Progress

## Overview

Implements Infisical Kubernetes Auth (Option C) — the operator now authenticates to Infisical using its own ServiceAccount JWT. No credentials are stored anywhere on-cluster.

## Changes Log

### 2026-03-02 17:14 - Infisical Kubernetes Auth (Option C)

**Problem**:
- Infisical operator was deployed but idle — no `InfisicalSecret` CRs existed and no auth method was configured
- Existing `infisical-secrets` k8s Secret was created by a bootstrap shell script (Option A/B: stored credentials)
- No RBAC allowing the operator SA to call the k8s `tokenreviews` API

**Solution**:
- Added `ClusterRole` (`infisical-token-reviewer`) granting `create` on `authentication.k8s.io/tokenreviews`
- Added `ClusterRoleBinding` binding that role to `infisical-operator-controller-manager` ServiceAccount in the `infisical` namespace
- Added `InfisicalSecret` CR (`infisical-bootstrap-secret`) configured with `kubernetesAuth`, referencing the operator's SA — operator authenticates using its own JWT, Infisical verifies via k8s tokenreviews callback, returns a short-lived access token
- Updated docs landing page (`_index.md`) to reflect Kubernetes Auth as the runtime secrets strategy

**Files Modified**:
- `workloads/secrets/infisical.go`
- `docs/content/_index.md`

**⚠️ One-time manual step required** before the CR becomes functional:

```bash
# 1. Deploy the changes
just synth
# (then ArgoCD applies the manifests including the new ClusterRole + binding)

# 2. Retrieve the token reviewer JWT (long-lived, for Infisical to call k8s tokenreviews)
kubectl create token infisical-operator-controller-manager \
  -n infisical --duration=8760h
```

Then in **Infisical UI → Access Control → Machine Identities → Create → "k8s-homelab"**:
- Kubernetes Host: `https://192.168.1.210:6443`
- Token Reviewer JWT: *(paste output from above command)*
- Allowed SA Names: `infisical-operator-controller-manager`
- Allowed Namespaces: `infisical`

Copy the generated `identityId` and replace `REPLACE_WITH_IDENTITY_ID` in `workloads/secrets/infisical.go`, then re-synth and push.

**Verification Steps**:
```bash
# Check CR was created
kubectl get infisicalsecret -n infisical

# Watch operator logs for successful kubernetesAuth handshake
kubectl logs -n infisical -l app.kubernetes.io/name=secrets-operator --tail=50

# Confirm synced secret was created
kubectl get secret infisical-synced-secrets -n infisical

# Inspect CR status conditions
kubectl describe infisicalsecret infisical-bootstrap-secret -n infisical
```

## Known Issues

### identityId placeholder must be filled manually

**Description**: The `InfisicalSecret` CR contains `REPLACE_WITH_IDENTITY_ID` as a placeholder. The operator will fail to authenticate until this is replaced with the real ID from the Infisical UI.

**Impact**: Secrets will not sync until the UI step is completed and the value is completed.

**Workaround**: The existing `infisical-secrets` bootstrap k8s Secret still exists and the Infisical application itself continues to run normally.

### 2026-03-05 12:16 - HTTPRoute ArgoCD Drift Fix + Harbor RWO PVC Deadlock

**Problem**:
- 5 apps (`comfyui`, `ollama`, `headlamp`, `rancher`, `infisical`) were `OutOfSync` in ArgoCD
- Root cause: HTTPRoute manifests omitted `group`/`kind` on `parentRefs` and `backendRefs`; the k8s API server defaulted them, causing ArgoCD to see a permanent diff
- `harbor` was `Degraded` — `jobservice` and `registry` pods stuck in `ContainerCreating` on `k8s-worker4` (RWO PVCs held by old pods on `k8s-worker1`, blocking rolling update)

**Solution**:
- Added `group: gateway.networking.k8s.io` + `kind: Gateway` to `parentRefs` in all 8 HTTPRoute Go sources
- Added `group: ""` + `kind: Service` to `backendRefs` in all 8 HTTPRoute Go sources
- Regenerated YAML manifests via `go run main.go` in `workloads/`
- Force-deleted old harbor pods holding RWO PVCs (`harbor-jobservice-64bd7fd9c8-n7wt2`, `harbor-registry-65d7d89c78-7ppgc`) — new pods immediately attached and came Running

**Files Modified**:

### 2026-03-05/06 - GPU Operator Validator Fix (Blackwell RTX 5070 Ti on Talos)

**Problem**:
- `comfyui` and `ollama` stuck `Pending` for 5+ days — `k8s-worker4` showing `nvidia.com/gpu: 0` allocatable
- `nvidia-operator-validator` stuck at `Init:0/4` for 22h+ causing device plugin to never advertise GPU
- Root causes (all specific to Talos pre-installed drivers + Blackwell GPU combo):
  1. `driver-validation` init: `nvidia-smi` not available as host binary on Talos (only in container overlays)
  2. `cuda-validation` init: CUDA sample failed with `forward compatibility on non-supported HW` — Blackwell RTX 5070 Ti requires CUDA 12.8 arch-native code; the validation sample targets older arch
  3. `plugin-validation` init: blocked behind cuda-validation, never reached
  4. Both `/run/nvidia/validations/driver-ready` AND `/run/nvidia/driver/driver-ready` required — bridge only created the latter

**Solution**:
- Removed `driver-validation`, `cuda-validation`, `plugin-validation` init containers from the validator DaemonSet via strategic merge patch (only `toolkit-validation` remains)
- Updated bridge DaemonSet to create `driver-ready` in **both** `/run/nvidia/validations/` AND `/run/nvidia/driver/`
- Added `.driver-ctr-ready` back to bridge (device plugin uses it as pre-installed driver signal)
- Set `driver.enabled=false` in ClusterPolicy and added `SKIP_VALIDATIONS=true` to validator env (belt-and-suspenders)

**Files Modified**:
- `workloads/hardware/nvidia_gpu_operator.go`

**Verification**:

# k8s-worker4   2   ← time-sliced (2 virtual GPUs from 1 physical RTX 5070 Ti)

### 2026-03-10/11 - Infisical → OpenBao Migration + n8n Chart Migration

#### OpenBao Migration

**Problem**:
- Infisical operator had multiple blocking issues: chart bugs (wrong namespace in RoleBindings), CRD schema errors breaking ArgoCD ServerSideApply, SA token discovery broken on k8s 1.24+, key features locked behind cloud plan
- Apps (`grafana`, `harbor`, `n8n`, `netbird`, `rancher`) all Degraded in ArgoCD post-migration

**Solution**:
- Replaced Infisical with **OpenBao** (Vault MPL-2.0 fork) + **Secrets Store CSI Driver**
- OpenBao unsealed via sidecar container reading `openbao-unseal-key` bootstrap Secret
- Two patterns:
  - **Pattern A** (Grafana): CSI file-only, `GF_SECURITY_ADMIN_PASSWORD__FILE=/mnt/secrets/ADMIN_PASSWORD`
  - **Pattern B** (Harbor, n8n, Rancher, NetBird): CSI secretObjects sync → k8s Secret → Helm `existingSecret`
- Harbor/Rancher: dedicated `secret-sync` Deployment (charts lack `extraVolumes` support)
- Grafana: removed Helm-managed `admin` Secret; set `GF_SECURITY_ADMIN_USER` via env
- n8n encryption key: stored in OpenBao, fetched via CSI, referenced via `existingEncryptionKeySecret`

**Files Modified**:

- HTTPRoute: `n8n-main:5678` → `n8n:80`
- **Note**: Deployment selector was immutable (label mismatch between old and new chart). Required manual `kubectl delete deployment n8n -n n8n` to unblock ArgoCD sync.
